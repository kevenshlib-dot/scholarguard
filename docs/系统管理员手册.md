## 1. 系统概述

### 1.1 产品简介

ScholarGuard 是一个基于大语言模型（LLM）的学术文本 AI 内容检测平台。系统采用"LLM-Center"架构，以 Qwen3.5-27B 等大模型为核心检测引擎，辅以轻量统计特征分析和证据融合机制，为高校和学术机构提供高精度的 AI 生成内容识别服务。

### 1.2 核心功能

| 功能模块 | 状态 | 说明 |
|---------|------|------|
| AI 文本检测 | **已上线** | 支持文档/段落/句子级检测，PDF/DOCX/TXT 上传 |
| 写作建议 | **已上线** | 基于检测结果智能推荐改写方案，一键优化 |
| 复核中心 | **已上线** | 高风险检测人工复核，支持维持/调整/撤销判定 |
| 翻译润色 | **待开发** | 学术文本翻译与润色（预留接口） |
| 系统管理 | **已上线** | 模型配置、公式调参、用户管理、审计日志 |

### 1.3 技术栈

- **后端：** Python 3.12 + FastAPI + SQLAlchemy (async) + Celery
- **前端：** React 18 + TypeScript + Tailwind CSS + Vite
- **数据库：** PostgreSQL 15
- **缓存/消息队列：** Redis 7.2
- **LLM 推理：** vLLM (Qwen3.5-27B) + LiteLLM 多模型路由
- **文件存储：** MinIO (S3 兼容)
- **向量数据库：** Qdrant（预留，暂未使用）
- **容器化：** Docker Compose

## 2. 系统架构

### 2.1 整体架构图

```
┌─────────────┐     ┌─────────────────────────────────────────┐
│   Browser    │────▶│        Frontend (React + Vite)           │
│  :3000       │     │    DetectPage / SuggestPage / Admin      │
└─────────────┘     └──────────────┬──────────────────────────┘
                                   │ HTTP (axios)
                                   ▼
                    ┌──────────────────────────────────┐
                    │   API Gateway (FastAPI + Uvicorn) │
                    │   :8000  /api/v1/*                │
                    │   ┌──────────────────────────┐   │
                    │   │ Auth │ Rate Limiter │ CORS│   │
                    │   └──────────────────────────┘   │
                    └────┬────────┬────────┬───────────┘
                         │        │        │
              ┌──────────┘   ┌────┘   ┌────┘
              ▼              ▼        ▼
     ┌──────────────┐  ┌─────────┐  ┌──────────────┐
     │  PostgreSQL   │  │  Redis  │  │ Celery Worker│
     │  :5433        │  │  :6379  │  │ (异步检测)    │
     └──────────────┘  └─────────┘  └──────┬───────┘
                                           │
                                           ▼
                                  ┌─────────────────┐
                                  │ vLLM / LiteLLM   │
                                  │ Qwen3.5-27B      │
                                  │ :8001             │
                                  └─────────────────┘
```

### 2.2 检测流水线

```
用户提交文本 → API 接收 (202 Accepted)
    → Celery Worker 异步处理：
        1. 文本预处理（分段、分句、统计）
        2. 缓存检查（Redis, SHA-256 哈希）
        3. LLM 评议 + 报告生成（单次合并调用）
        4. 轻量统计特征分析
        5. 证据融合 → 风险分数 + 等级
        6. 结果写入 DB + 缓存
    → 前端轮询 GET /detect/{task_id}
    → 展示结果（风险等级、高亮标注、证据摘要）
```

docker compose up -d

**各服务端口映射：**

|------|---------|-----------|------|
| frontend | 3000 | 3000 | 用户界面 |

### 3.4 本地开发部署

```bash
# 1. 启动基础设施
docker compose up -d postgres redis

# 2. 后端 API
cd api
uv sync                    # 安装 Python 依赖
uv run alembic upgrade head  # 数据库迁移
uv run uvicorn app.main:create_app --factory --host 0.0.0.0 --port 8000 --reload

# 3. Celery Worker（新终端）
cd api
uv run celery -A app.worker worker --loglevel=info --concurrency=1

# 4. 前端（新终端）
cd frontend
npm install
npm run dev                # 默认 http://localhost:3000
```

### 3.5 数据库初始化

系统首次启动时自动执行 Alembic 迁移创建表结构。如需手动迁移：

```bash
cd api
uv run alembic upgrade head     # 升级到最新版本
uv run alembic downgrade -1     # 回退一个版本
uv run alembic history          # 查看迁移历史
```

## 4. 服务组件详解

### 4.1 FastAPI 后端 (api)

- **框架：** FastAPI + Uvicorn (ASGI)
- **入口：** `api/app/main.py` → `create_app()` 工厂函数
- **启动生命周期：** 初始化数据库连接池 → Redis 连接 → 注册中间件 → 注册路由
- **中间件栈：** CORS → TrustedHost → GZip → 自定义异常处理
- **路由前缀：** `/api/v1`

### 4.2 Celery Worker

- **Broker：** Redis (`redis://localhost:6379/0`)
- **Result Backend：** Redis
- **注册任务：**
  - `app.tasks.run_detection` — AI 内容检测
  - `app.tasks.run_heatmap` — 段落级热力图生成
- **并发模式：** prefork（默认 2 个 worker 进程）

### 4.3 PostgreSQL 数据库

- **版本：** PostgreSQL 15
- **数据库名：** `scholarguard`
- **用户名：** `sguser`
- **密码：** `sgpass_dev`（开发环境，生产环境请修改）
- **端口：** 5433（宿主机） → 5432（容器内）
- **ORM：** SQLAlchemy 2.0 (async, asyncpg 驱动)

### 4.4 Redis

- **版本：** Redis 7.2 Alpine
- **用途：**
  - Celery 消息队列 broker
  - 检测结果缓存（24小时 TTL, key: `sg:detect:{content_hash}`）
  - 速率限制令牌桶
- **端口：** 6379

### 4.5 MinIO 对象存储

- **用途：** 上传文件存储（PDF、DOCX 等）
- **端口：** 9000 (API) / 9001 (管理控制台)
- **默认凭证：** `minioadmin` / `minioadmin`
- **Bucket：** `scholarguard`

### 4.6 Qdrant 向量数据库

- **用途：** 预留用于语义检索（文献研究功能，当前未启用）
- **端口：** 6333 (HTTP) / 6334 (gRPC)

### 4.7 vLLM 推理服务

- **部署位置：** 独立 GPU 服务器 `192.168.31.18:8001`
- **模型：** Qwen3.5-27B
- **接口：** OpenAI 兼容 API (`/v1/chat/completions`)
- **通过 LiteLLM 统一调用，** 支持自动回退到其他模型

## 6. 账号与凭证管理

### 6.1 认证方式

系统支持两种认证方式：

#### 6.1.1 JWT Bearer Token

- **获取方式：** `POST /api/v1/auth/login` 返回 `access_token`
- **使用方式：** HTTP Header `Authorization: Bearer <token>`
- **有效期：** 60 分钟（可配置）
- **令牌内容：** `{sub: user_id, role: role, exp: expiry, email: email}`

#### 6.1.2 API Key（服务账号）

- **使用方式：** HTTP Header `X-API-Key: <key>`
- **当前配置的 Key：**
  - `sg-test-key-001` — 测试用途
  - `sg-dev-key-002` — 开发用途
- **权限：** `service` 角色，拥有大部分 API 访问权
- **生产环境应删除测试 key 并生成新 key**

### 6.2 用户角色

| 角色 | 代码 | 权限说明 |
|------|------|---------|
| 管理员 | `admin` | 全部权限，包括系统管理、用户管理、公式调参 |
| 超级管理员 | `super_admin` | 同管理员 |
| 机构管理员 | `org_admin` | 机构级管理 |
| 检测员 | `detector` | 提交检测、查看结果（默认注册角色） |
| 审核员 | `reviewer` / `instructor` | 人工复核检测结果 |
| 审计员 | `auditor` | 查看审计日志 |
| API 调用者 | `api_caller` | 仅 API 调用 |
| 服务账号 | `service` | API Key 认证用户的默认角色 |

### 6.3 默认账号

| 用途 | 类型 | 凭证 |
|------|------|------|
| 开发测试 | API Key | `X-API-Key: sg-test-key-001` |
| 开发测试2 | API Key | `X-API-Key: sg-dev-key-002` |
| PostgreSQL | 数据库 | 用户 `sguser` / 密码 `sgpass_dev` |
| MinIO 控制台 | 存储管理 | 用户 `minioadmin` / 密码 `minioadmin` |

> 新用户通过 `POST /api/v1/auth/register` 注册，默认角色为 `detector`。管理员可通过管理面板更改用户角色。

  "text": "待检测的学术文本...",

## 10. 检测公式与参数调优

### 10.1 风险分数计算公式

```
risk_score = w1 × llm_confidence + w2 × statistical_score + w3 × stylistic_score
```

### 10.2 默认权重参数

| 参数 | 含义 | 默认值 | 范围 |
|------|------|--------|------|
| `w_llm_confidence` | LLM 置信度权重 | 0.6 | 0-1 |
| `w_statistical_score` | 统计特征权重 | 0.25 | 0-1 |
| `w_stylistic_score` | 风格特征权重 | 0.15 | 0-1 |

### 10.3 风险等级阈值

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `t_low_max` | 低风险上限 | 0.3 |
| `t_medium_max` | 中风险上限 | 0.6 |
| `t_high_max` | 高风险上限 | 0.85 |

**等级划分：**
- **低风险 (low)：** risk_score ≤ 0.3 — 自然人类写作
- **中风险 (medium)：** 0.3 < risk_score ≤ 0.6 — 部分 AI 特征
- **高风险 (high)：** 0.6 < risk_score ≤ 0.85 — 明显 AI 特征
- **极高风险 (critical)：** risk_score > 0.85 — 确定 AI 生成

### 10.4 参数调整方法

通过系统管理 → 公式参数页面，或 API：

```bash
# 获取当前参数
curl -H "X-API-Key: sg-test-key-001" \
  http://localhost:8000/api/v1/admin/formula-params

# 更新参数
curl -X PUT -H "X-API-Key: sg-test-key-001" \
  -H "Content-Type: application/json" \
  -d '{"weights":{"llm_confidence":0.65,"statistical_score":0.2,"stylistic_score":0.15},"thresholds":{"low_max":0.3,"medium_max":0.55,"high_max":0.8}}' \
  http://localhost:8000/api/v1/admin/formula-params
```

## 11. 速率限制策略

### 11.1 令牌桶算法

系统使用 Redis 支撑的令牌桶算法进行速率限制，基于 Lua 脚本原子操作。

### 11.2 角色级限制

| 角色 | 桶容量 | 补充速率 | 等效限制 |
|------|--------|---------|---------|
| user（普通用户） | 30 | 0.5/秒 | 约每2秒1次请求，突发30次 |
| service（服务账号） | 120 | 2.0/秒 | 约每0.5秒1次，突发120次 |
| admin（管理员） | 300 | 5.0/秒 | 约每0.2秒1次，突发300次 |

### 11.3 特殊端点限制

| 端点 | 桶容量 | 补充速率 | 说明 |
|------|--------|---------|------|
| `POST /detect/batch` | 5 | 0.1/秒 | 批量检测严格限制 |

### 11.4 响应头

被限制时返回 `429 Too Many Requests`，响应头包含：
- `Retry-After: <秒>` — 下次可请求的等待时间
- `X-RateLimit-Remaining: <数量>` — 剩余令牌数

## 12. 异步任务系统

### 12.1 Celery 配置

```python
# 任务注册
app.tasks.run_detection    # AI 内容检测
app.tasks.run_heatmap      # 段落热力图生成
```

### 12.2 检测任务流程

1. API 收到请求 → 创建 `DetectionResult`（status=pending）→ 返回 202
2. `run_detection.delay()` 派发到 Celery
3. Worker 执行：预处理 → LLM 调用 → 统计分析 → 证据融合
4. 更新 DB 记录 status=completed，写入所有结果字段
5. 结果写入 Redis 缓存（24h TTL）

### 12.3 重要注意事项

- **Celery Worker 是独立进程，** 代码修改后需要手动重启
- **重启命令：**
  ```bash
  # 找到并终止旧进程
  ps aux | grep celery | grep -v grep | awk '{print $2}' | xargs kill -9
  # 启动新进程
  cd api && uv run celery -A app.worker worker --loglevel=info --concurrency=1
  ```

## 17. 故障排查

### 17.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 前端白屏 | Vite 开发服务器崩溃 | `kill` 进程后重启 `npm run dev` |
| API 返回 500 | 数据库连接断开 | 检查 Docker 是否运行：`docker compose ps` |
| 检测一直 pending | Celery Worker 未运行 | 重启 Worker（见 12.3） |
| 检测超时 | vLLM 服务不可达 | 检查 GPU 服务器 `192.168.31.18:8001` 连通性 |
| 429 Too Many Requests | 触发速率限制 | 等待 `Retry-After` 秒后重试 |
| 404 Not Found | 端点不存在或路径错误 | 检查 API 文档确认正确路径 |
| Docker 服务掉线 | Docker Desktop 关闭 | 打开 Docker Desktop → `docker compose up -d` |
| 高亮文本不显示 | Celery Worker 使用旧 Prompt | 重启 Celery Worker |

tail -f /tmp/uvicorn_new.log

# Celery 日志
tail -f /tmp/celery_worker.log

# Docker 日志
