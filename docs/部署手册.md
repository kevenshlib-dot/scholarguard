## 1. 部署概述

### 1.1 部署模式

ScholarGuard 支持两种主要部署模式：

| 模式 | 适用场景 | 说明 |
|------|---------|------|
| **Docker Compose 全容器化** | 推荐方式，适用于开发与生产 | 所有服务运行在 Docker 容器中，一键启动 |
| **混合部署** | 本地开发调试 | 基础设施运行在 Docker 中，应用服务运行在本地 |

### 1.2 服务组成

整个系统由以下 **7 个服务** 组成：

```
┌─────────────────────────────────────────────────────┐
│                   ScholarGuard 服务架构               │
├─────────────────────────────────────────────────────┤
│                                                     │
│  基础设施层：                                         │
│  ┌──────────┐ ┌───────┐ ┌────────┐ ┌───────┐       │
│  │PostgreSQL│ │ Redis │ │ Qdrant │ │ MinIO │       │
│  │  :5433   │ │ :6379 │ │ :6333  │ │ :9000 │       │
│  └────┬─────┘ └───┬───┘ └───┬────┘ └───┬───┘       │
│       │           │         │           │           │
│  应用层：                                            │
│  ┌────┴───────────┴─────────┴───────────┘           │
│  │                                                  │
│  │  ┌──────────────┐  ┌────────────────┐            │
│  │  │  FastAPI API  │  │ Celery Worker  │            │
│  │  │    :8000      │  │  (后台进程)     │            │
│  │  └──────┬───────┘  └────────────────┘            │
│  │         │                                        │
│  │  ┌──────┴───────┐                                │
│  │  │   Frontend   │                                │
│  │  │ (Nginx:3000) │                                │
│  │  └──────────────┘                                │
│  │                                                  │
│  外部依赖：                                          │
│  ┌──────────────────────────┐                       │
│  │  vLLM / Ollama 模型服务   │                       │
│  │  (192.168.31.18:8001)    │                       │
│  └──────────────────────────┘                       │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 1.3 端口规划

| 服务 | 容器内端口 | 宿主机端口 | 协议 | 说明 |
|------|-----------|-----------|------|------|
| PostgreSQL | 5432 | **5433** | TCP | 避免与本地 PostgreSQL 冲突 |
| Redis | 6379 | **6379** | TCP | 缓存 / 消息队列 / 速率限制 |
| Qdrant HTTP | 6333 | **6333** | HTTP | 向量数据库 REST API |
| Qdrant gRPC | 6334 | **6334** | gRPC | 向量数据库 gRPC 接口 |
| MinIO API | 9000 | **9000** | HTTP | S3 兼容对象存储 API |
| MinIO Console | 9001 | **9001** | HTTP | MinIO 管理控制台 |
| FastAPI | 8000 | **8000** | HTTP | 后端 API 服务 |
| Frontend | 80/3000 | **3000** | HTTP | 前端页面 |
| vLLM（外部） | 8001 | **8001** | HTTP | LLM 推理服务 |

## 2. 环境要求

### 2.1 硬件要求

| 项目 | 最低要求 | 推荐配置 |
|------|---------|---------|
| CPU | 4 核 | 8 核以上 |
| 内存 | 8 GB | 16 GB 以上 |
| 磁盘 | 20 GB SSD | 100 GB SSD |
| GPU | 不必需（使用远程 vLLM） | NVIDIA GPU（若本地运行 LLM） |

| Python | 3.12+ | 后端运行时 |
| uv | 最新版 | Python 包管理器 |
| Node.js | 20+ | 前端运行时 |

```
                             ▼
                    ┌─────────────────┐
                    │   FastAPI API   │

                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │PostgreSQL│  │  Redis   │  │  Celery  │
        │ 关系数据  │  │ 缓存/队列 │  │ 异步检测  │
        └──────────┘  └──────────┘  └────┬─────┘

                                         ▼
```

### 3.2 服务依赖关系

```
postgres ─────┐
              ├──→ api ──→ frontend

postgres ─────┐
              ├──→ celery_worker
redis ────────┘
```

启动顺序由 Docker Compose 的 `depends_on` + `healthcheck` 自动管理。

## 4. 环境变量配置

### 4.1 配置文件

项目根目录下有两个配置文件：

| 文件 | 说明 |
|------|------|
| `.env.example` | 配置模板，包含所有可配置项及说明 |
| `.env` | 实际使用的配置文件（不纳入版本控制） |

### 4.2 首次配置

```bash
# 从模板复制配置文件
cp .env.example .env

# 编辑配置
vi .env    # 或使用你喜欢的编辑器
```

### 4.3 完整环境变量参考

#### 数据库

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `DATABASE_URL` | `postgresql+asyncpg://sguser:sgpass_dev@postgres:5432/scholarguard` | 数据库连接字符串 |

> ⚠️ Docker Compose 内部使用 `postgres:5432`，本地开发使用 `localhost:5433`

**Docker 内连接（docker-compose.yml 中已覆盖）：**
```
DATABASE_URL=postgresql+asyncpg://sguser:sgpass_dev@postgres:5432/scholarguard
```

**本地开发连接（.env 文件中设置）：**
```
DATABASE_URL=postgresql+asyncpg://sguser:sgpass_dev@localhost:5433/scholarguard
```

#### 缓存与消息队列

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `REDIS_URL` | `redis://redis:6379/0` | Redis 连接地址，同时用于缓存和 Celery 消息队列 |

#### 对象存储（MinIO）

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `MINIO_ENDPOINT` | `minio:9000` | MinIO 服务地址 |
| `MINIO_ACCESS_KEY` | `minioadmin` | 访问密钥 |
| `MINIO_SECRET_KEY` | `minioadmin` | 私钥 |
| `MINIO_BUCKET` | `scholarguard` | 默认存储桶名 |
| `MINIO_USE_SSL` | `false` | 是否启用 SSL |

#### 向量数据库

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `QDRANT_URL` | `http://qdrant:6333` | Qdrant 服务地址 |

#### JWT 认证

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `JWT_SECRET_KEY` | `CHANGE_ME_TO_A_RANDOM_SECRET` | JWT 签名密钥 **⚠️生产环境必须修改** |
| `JWT_ALGORITHM` | `HS256` | JWT 签名算法 |
| `JWT_EXPIRE_MINUTES` | `60` | Token 过期时间（分钟） |

**生成安全的 JWT 密钥：**
```bash
python3 -c "import secrets; print(secrets.token_urlsafe(64))"
```

#### LLM 模型后端

| 变量 | 默认值 | 说明 |

| `OLLAMA_URL` | `http://ollama:11434` | Ollama 本地模型服务地址 |
| `OPENAI_API_KEY` | （空） | OpenAI API Key（可选） |
| `ANTHROPIC_API_KEY` | （空） | Anthropic API Key（可选） |
| `GOOGLE_API_KEY` | （空） | Google AI API Key（可选） |

#### 应用配置

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `LOG_LEVEL` | `INFO` | 日志级别：DEBUG / INFO / WARNING / ERROR |
| `ENVIRONMENT` | `development` | 运行环境标识 |
| `FORMULA_VERSION` | `v1.0` | 评分公式版本 |
| `PARAM_VERSION` | `v1.1` | 参数版本 |
| `ALLOWED_API_KEYS` | `[]` | 允许的 API Key 列表（JSON 数组格式） |

### 4.4 开发环境 vs 生产环境配置差异

| 配置项 | 开发环境 | 生产环境 |
|-------|---------|---------|
| `DATABASE_URL` 主机 | `localhost:5433` 或 `postgres:5432` | 独立数据库服务器地址 |
| `JWT_SECRET_KEY` | 默认值可用 | **必须使用强随机字符串** |
| `LOG_LEVEL` | `DEBUG` 或 `INFO` | `WARNING` 或 `ERROR` |
| `ENVIRONMENT` | `development` | `production` |
| `MINIO_ACCESS_KEY/SECRET` | `minioadmin` | **必须使用强密码** |
| CORS 来源 | `["*"]`（代码中） | 限制为具体域名 |

## 5. Docker Compose 一键部署

### 5.1 获取代码

```bash
git clone <仓库地址> scholarguard
cd scholarguard
```

### 5.2 准备配置

```bash
# 复制环境配置
cp .env.example .env

# 按需修改配置（尤其是生产环境）
vi .env
```

### 5.3 启动所有服务

```bash
# 启动全部服务（后台运行）
docker compose up -d

# 查看服务状态
docker compose ps
```

预期输出：
```
NAME                    STATUS              PORTS
scholarguard-postgres   Up (healthy)        0.0.0.0:5433->5432/tcp
scholarguard-redis      Up (healthy)        0.0.0.0:6379->6379/tcp
scholarguard-qdrant     Up (healthy)        0.0.0.0:6333-6334->6333-6334/tcp
scholarguard-minio      Up                  0.0.0.0:9000-9001->9000-9001/tcp
scholarguard-api        Up                  0.0.0.0:8000->8000/tcp
scholarguard-celery     Up
scholarguard-frontend   Up                  0.0.0.0:3000->3000/tcp
```

### 5.4 初始化数据库（首次部署）

```bash
# 进入 API 容器执行数据库迁移
docker compose exec api bash -c "alembic upgrade head"
```

### 5.5 验证部署

```bash
# 检查 API 健康状态
curl http://localhost:8000/health

# 预期返回
# {"status":"healthy","version":"0.1.0","environment":"development"}

# 访问前端
open http://localhost:3000    # macOS
xdg-open http://localhost:3000  # Linux
```

### 5.6 停止服务

```bash
# 停止所有服务（保留数据）
docker compose down

# 停止并删除数据卷（⚠️ 会丢失所有数据）
docker compose down -v
```

### 5.7 Docker Compose 文件说明

完整的 `docker-compose.yml` 包含以下服务定义：

```yaml
version: "3.9"

services:
  # ─── 基础设施 ───
  postgres:        # PostgreSQL 15 数据库
  redis:           # Redis 7.2 缓存/队列
  qdrant:          # Qdrant 向量数据库
  minio:           # MinIO 对象存储

  # ─── 应用服务 ───
  api:             # FastAPI 后端 (依赖: postgres, redis, qdrant, minio)
  celery_worker:   # Celery 工作进程 (依赖: postgres, redis)
  frontend:        # React 前端 via Nginx (依赖: api)

volumes:
  pg_data:         # PostgreSQL 数据持久化
  redis_data:      # Redis 数据持久化
  qdrant_data:     # Qdrant 索引持久化
  minio_data:      # MinIO 文件持久化
```

### 5.8 命名卷说明

| 卷名 | 挂载路径 | 用途 | 注意事项 |
|------|---------|------|---------|
| `pg_data` | `/var/lib/postgresql/data` | 数据库文件 | 包含所有业务数据，务必备份 |
| `redis_data` | `/data` | Redis RDB/AOF 持久化 | 缓存数据，丢失可恢复 |
| `qdrant_data` | `/qdrant/storage` | 向量索引 | 语义搜索数据 |
| `minio_data` | `/data` | 上传文件存储 | 用户上传的文档 |

## 6. 基础设施服务详解

### 6.1 PostgreSQL 15

**用途：** 主关系型数据库，存储用户、检测结果、复核记录等所有业务数据。

| 项目 | 值 |
|------|-----|
| 镜像 | `postgres:15` |
| 宿主机端口 | 5433 |
| 数据库名 | `scholarguard` |
| 用户名 | `sguser` |
| 密码 | `sgpass_dev`（开发环境） |

**健康检查：**
```bash
pg_isready -U sguser -d scholarguard
```

**手动连接：**
```bash
# 通过 Docker
docker compose exec postgres psql -U sguser -d scholarguard

# 通过宿主机
psql -h localhost -p 5433 -U sguser -d scholarguard
```

**连接池配置（应用端）：**
```
pool_size = 20          # 常驻连接数
max_overflow = 10       # 突发可额外创建的连接数
pool_pre_ping = True    # 使用前验证连接有效性
```

### 6.2 Redis 7.2

**用途：** 多重角色 — 缓存层、Celery 消息队列、速率限制器、检测结果临时存储。

| 项目 | 值 |
|------|-----|
| 镜像 | `redis:7.2-alpine` |
| 宿主机端口 | 6379 |
| 数据库编号 | 0 |

**健康检查：**
```bash
redis-cli ping
# 返回 PONG
```

**常用调试命令：**
```bash
# 进入 Redis CLI
docker compose exec redis redis-cli

# 查看所有键
KEYS *

# 查看速率限制令牌桶
GET rl:user:<user_id>

# 查看 Celery 队列长度
LLEN celery

# 查看内存使用
INFO memory
```

### 6.3 Qdrant

**用途：** 向量数据库，用于语义搜索和文献相似度比对。

| 项目 | 值 |
|------|-----|
| 镜像 | `qdrant/qdrant:latest` |
| HTTP 端口 | 6333 |
| gRPC 端口 | 6334 |

**健康检查：**
```bash
curl http://localhost:6333/healthz
```

**管理界面：**
```
http://localhost:6333/dashboard
```

### 6.4 MinIO

**用途：** S3 兼容对象存储，用于存储用户上传的文档文件。

| 项目 | 值 |
|------|-----|
| 镜像 | `minio/minio:latest` |

| 管理员用户 | `minioadmin` |
| 管理员密码 | `minioadmin` |

## 7. 后端 API 服务部署

### 7.1 Dockerfile 构建策略

API 使用**三阶段 Docker 构建**：

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Stage 1:       │     │  Stage 2:       │     │  Stage 3:       │
│  builder        │────→│  development    │     │  production     │
│                 │     │                 │     │                 │
│  - Python 3.12  │     │  - 含 builder   │     │  - 含 builder   │
│  - 安装 uv      │     │    依赖         │     │    依赖         │
│  - 安装依赖到   │     │  - 安装 curl    │     │  - 非root用户   │
│    .venv       │     │  - hot-reload   │     │  - 4 workers    │
└─────────────────┘     │  - 绑定挂载源码 │     │  - 无 reload    │
                        └─────────────────┘     └─────────────────┘
```

| 构建目标 | 用途 | 启动命令 |
|---------|------|---------|
| `development` | 本地开发，支持热重载 | `uvicorn main:app --reload` |
| `production` | 生产部署，多 Worker | `uvicorn main:app --workers 4` |

### 7.2 docker-compose 中的 API 配置

```yaml
api:
  build:
    context: ./api
    dockerfile: Dockerfile
    target: development        # 开发模式，生产改为 production
  ports:
    - "8000:8000"
  env_file:
    - .env
  environment:                 # Docker 内部覆盖，使用容器名
    DATABASE_URL: postgresql+asyncpg://sguser:sgpass_dev@postgres:5432/scholarguard
    REDIS_URL: redis://redis:6379/0
    MINIO_ENDPOINT: minio:9000
    QDRANT_URL: http://qdrant:6333
  volumes:
    - ./api:/app               # 绑定挂载源码，热重载
    - /app/.venv               # 匿名卷，防止宿主机 .venv 覆盖容器
  depends_on:
    postgres: { condition: service_healthy }
    redis:    { condition: service_healthy }
    qdrant:   { condition: service_healthy }
    minio:    { condition: service_started }
```

### 7.3 应用启动流程

FastAPI 应用使用 **lifespan** 管理启动和关闭：

**启动阶段：**
1. 加载环境配置（pydantic-settings 从 .env 读取）
2. 配置日志系统
3. 建立 Redis 连接池（失败不阻塞启动）
4. 初始化 SQLAlchemy 异步引擎和会话工厂
5. 打印启动信息（环境、公式版本、参数版本）

**关闭阶段：**
1. 关闭数据库引擎，释放连接池
2. 关闭 Redis 连接池

### 7.4 中间件栈

| 中间件 | 顺序 | 功能 |
|-------|------|------|
| CORS | 1 | 跨域资源共享 |
| Request-ID | 2 | 为每个请求生成唯一 UUID (X-Request-ID) |
| 耗时计算 | 3 | 计算请求处理时间 (X-Processing-Time-Ms) |
| 速率限制 | 4 | 基于 Redis 令牌桶的请求限流 |

### 7.5 API 路由挂载

## 8. Celery 异步任务服务部署

### 8.1 作用

Celery Worker 负责执行**耗时的 AI 检测任务**。当用户提交检测请求时，API 服务将任务放入 Redis 队列，Celery Worker 从队列中取出并异步执行，包括调用 LLM 推理、统计分析、评分计算等。

### 8.2 配置

```yaml
celery_worker:
  build:
    context: ./api
    dockerfile: Dockerfile
    target: development
  command: ["celery", "-A", "app.worker", "worker", "--loglevel=info", "--concurrency=2"]
  env_file:
    - .env
  environment:
    DATABASE_URL: postgresql+asyncpg://sguser:sgpass_dev@postgres:5432/scholarguard
    REDIS_URL: redis://redis:6379/0
  depends_on:
    postgres: { condition: service_healthy }
    redis:    { condition: service_healthy }
```

### 8.3 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `-A app.worker` | — | Celery 应用模块路径 |
| `--loglevel` | `info` | 日志级别 |
| `--concurrency` | `2` | 并发 Worker 进程数 |
| `task_serializer` | `json` | 任务序列化格式 |
| `task_acks_late` | `True` | 任务完成后才确认，确保不丢失 |
| `worker_prefetch_multiplier` | `1` | 每次只预取一个任务，保证负载均衡 |
| `result_expires` | `86400` (24h) | 任务结果过期时间 |

### 8.4 任务生命周期

```
用户提交检测
    ↓
API 创建 DetectionResult 记录 (status="pending")
    ↓
API 发送任务到 Redis 队列
    ↓
Celery Worker 取出任务 (status="processing")
    ↓
调用 LLM 推理 + 统计分析 + 评分计算
    ↓
写入结果到 PostgreSQL (status="completed")
    ↓
前端轮询 GET /detect/{task_id} 获取结果
```

### 8.5 重启 Celery Worker

当修改了检测相关代码（如 prompt 模板、检测引擎逻辑）后，**必须重启 Celery Worker** 才能生效：

```bash
# Docker 方式
docker compose restart celery_worker

# 本地方式
# 先找到并终止旧进程
ps aux | grep celery | grep -v grep
kill <PID>

# 重新启动
cd api
source .venv/bin/activate
celery -A app.worker worker --loglevel=info --concurrency=2
```

> ⚠️ **重要提示**：Celery Worker 是独立进程，与 FastAPI 热重载无关。修改 prompt 或检测逻辑后如果忘记重启 Worker，会导致旧的代码继续运行。

```nginx
    listen 80;
    server_name _;

    index index.html;

    # API 反向代理 — 将 /api/ 请求转发到后端

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        try_files $uri $uri/ /index.html;
```

## 11. LLM 模型服务配置

### 11.1 架构设计

ScholarGuard 使用 **LiteLLM** 作为统一 LLM 网关，支持多种模型提供者的无缝切换：

```
ScholarGuard API
    │
    ▼
┌──────────┐
│ LiteLLM  │ ──→ vLLM (Qwen3.5-27B) [主力]
│  Gateway │ ──→ Ollama (本地模型)   [备选]
│          │ ──→ OpenAI API          [外部]
│          │ ──→ Anthropic API       [外部]
│          │ ──→ Google AI API       [外部]
└──────────┘
```

### 11.2 vLLM 部署（推荐）

vLLM 是当前系统的**主力推理后端**，运行 Qwen3.5-27B 模型。

**当前配置：**
```
VLLM_URL=http://192.168.31.18:8001/v1
模型: Qwen/Qwen3.5-27B
```

**vLLM 服务器启动（GPU 服务器上执行）：**
```bash
# 安装 vLLM
pip install vllm

# 启动服务（需 NVIDIA GPU）
python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen3.5-27B \
    --host 0.0.0.0 \
    --port 8001 \
    --tensor-parallel-size 1 \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.9
```

**GPU 要求：**
- Qwen3.5-27B：至少 24GB VRAM（如 RTX 4090 或 A100）
- 推荐 40GB+ VRAM 以支持较长上下文

### 11.3 Ollama 部署（备选）

Ollama 适用于资源有限的环境或快速原型验证：

**在 docker-compose.yml 中取消注释 ollama 服务：**
```yaml
ollama:
  image: ollama/ollama:latest
  restart: unless-stopped
  ports:
    - "11434:11434"
  volumes:
    - ollama_models:/root/.ollama
  deploy:
    resources:
      reservations:
        devices:
          - driver: nvidia
            count: 1
            capabilities: [gpu]
```

**拉取模型：**
```bash
docker compose exec ollama ollama pull qwen2.5:14b
```

**配置环境变量：**
```
OLLAMA_URL=http://ollama:11434
```

### 11.4 三级降级策略

系统内置 LLM 调用降级机制：

```
第 1 级: 主力模型 (vLLM / Qwen3.5-27B)
    ↓ 失败
第 2 级: 备用模型 (Ollama / 其他在线模型)
    ↓ 失败
第 3 级: 降级模式 (仅统计特征分析，无 LLM)
```

```yaml
api:
  logging:
    driver: "json-file"
    options:
      max-size: "50m"
      max-file: "5"
```

或者将日志输出到宿主机文件：
```yaml
volumes:
  - ./logs/api:/app/logs
```

## 17. 备份与恢复

### 17.1 数据库备份

**手动全量备份：**
```bash
# 导出 SQL 格式
docker compose exec -T postgres pg_dump -U sguser scholarguard > backup_$(date +%Y%m%d_%H%M%S).sql

# 导出压缩格式（推荐）
docker compose exec -T postgres pg_dump -U sguser -Fc scholarguard > backup_$(date +%Y%m%d_%H%M%S).dump
```

**定时备份（crontab）：**
```bash
# 每天凌晨 2 点执行
0 2 * * * cd /path/to/scholarguard && docker compose exec -T postgres pg_dump -U sguser -Fc scholarguard > /backups/db_$(date +\%Y\%m\%d).dump
```

**恢复数据库：**
```bash
# 从 SQL 文件恢复
docker compose exec -T postgres psql -U sguser -d scholarguard < backup.sql

# 从 dump 文件恢复
docker compose exec -T postgres pg_restore -U sguser -d scholarguard < backup.dump
```

### 17.2 Redis 数据备份

Redis 的数据主要是缓存和临时队列，通常不需要备份。如需备份：

```bash
# 触发 RDB 快照
docker compose exec redis redis-cli BGSAVE

# 复制 RDB 文件
docker cp scholarguard-redis-1:/data/dump.rdb ./redis_backup.rdb
```

### 17.3 MinIO 文件备份

```bash
# 使用 MinIO Client (mc) 镜像同步
docker compose exec minio mc alias set local http://localhost:9000 minioadmin minioadmin
docker compose exec minio mc mirror local/scholarguard /backup/minio/
```

或直接备份 Docker 卷：
```bash
docker run --rm -v scholarguard_minio_data:/data -v $(pwd):/backup alpine \
    tar czf /backup/minio_backup.tar.gz -C /data .
```

### 17.4 完整备份方案

```bash
#!/bin/bash
# full_backup.sh — ScholarGuard 全量备份脚本

BACKUP_DIR="/backups/scholarguard/$(date +%Y%m%d_%H%M%S)"
mkdir -p "$BACKUP_DIR"

echo "📦 开始全量备份..."

# 1. 数据库
echo "  → 备份 PostgreSQL..."
docker compose exec -T postgres pg_dump -U sguser -Fc scholarguard > "$BACKUP_DIR/database.dump"

# 2. Redis RDB
echo "  → 备份 Redis..."
docker compose exec -T redis redis-cli BGSAVE
sleep 2
docker cp scholarguard-redis-1:/data/dump.rdb "$BACKUP_DIR/redis.rdb"

# 3. 环境配置
echo "  → 备份配置..."
cp .env "$BACKUP_DIR/env.bak"

# 4. 压缩
echo "  → 压缩备份文件..."
tar czf "$BACKUP_DIR.tar.gz" -C "$(dirname $BACKUP_DIR)" "$(basename $BACKUP_DIR)"
rm -rf "$BACKUP_DIR"

echo "✅ 备份完成: $BACKUP_DIR.tar.gz"
```

## 19. 常见部署问题排查

### 19.1 服务无法启动

**问题：** `docker compose up` 后某个服务一直重启

**排查步骤：**
```bash
# 1. 查看服务状态
docker compose ps

# 2. 查看问题服务的日志
docker compose logs <service_name>

# 3. 常见原因：
#    - 端口被占用：lsof -i :<port>
#    - 配置错误：检查 .env 文件
#    - 依赖服务未就绪：检查 healthcheck 状态
```

### 19.2 数据库连接失败

**错误：** `connection refused` 或 `could not connect to server`

**解决：**
```bash
# 检查 PostgreSQL 状态
docker compose ps postgres
docker compose logs postgres

# 确认端口映射
# Docker 内部: postgres:5432
# 本地访问:   localhost:5433 (注意不是 5432!)

# 确认 DATABASE_URL 格式正确
# Docker 内: postgresql+asyncpg://sguser:sgpass_dev@postgres:5432/scholarguard
# 本地:      postgresql+asyncpg://sguser:sgpass_dev@localhost:5433/scholarguard
```

### 19.3 端口冲突

**错误：** `Bind for 0.0.0.0:XXXX failed: port is already allocated`

**解决：**
```bash
# 查找占用端口的进程
lsof -i :3000    # 前端
lsof -i :8000    # API
lsof -i :5433    # PostgreSQL
lsof -i :6379    # Redis

# 终止占用进程
kill <PID>

# 或修改 docker-compose.yml 中的端口映射
```

### 19.4 Celery Worker 不执行任务

**现象：** 提交检测后一直显示 "pending"

**排查：**
```bash
# 1. 确认 Worker 在运行
docker compose ps celery_worker
docker compose logs celery_worker

# 2. 检查 Redis 连接
docker compose exec redis redis-cli ping

# 3. 检查队列中是否有任务
docker compose exec redis redis-cli LLEN celery

# 4. 重启 Worker
docker compose restart celery_worker
```

### 19.5 LLM 调用超时

**现象：** 检测任务失败，错误信息含 "timeout" 或 "connection refused"

**排查：**
```bash
# 1. 确认 vLLM 服务可达
curl http://192.168.31.18:8001/v1/models

# 2. 如果不可达，检查：
#    - vLLM 服务器是否在运行
#    - 网络是否互通（防火墙、VPN）
#    - GPU 是否有足够内存

# 3. 系统会自动降级到统计特征分析模式
```

### 19.6 前端显示空白页

**可能原因：**
1. API 服务未启动 → 检查 `http://localhost:8000/health`
2. 代理配置错误 → 确认 Vite/Nginx 的 proxy 配置
3. 构建失败 → 查看 `docker compose logs frontend`

### 19.7 Docker Desktop 崩溃后恢复

```bash
# 1. 重新打开 Docker Desktop
open -a Docker    # macOS
# 或重启 Docker 服务
sudo systemctl restart docker  # Linux

# 2. 等待 Docker 完全启动后重新启动服务
docker compose up -d

# 3. 验证所有服务
docker compose ps
curl http://localhost:8000/health
```

## 20. 升级与回滚

### 20.1 版本升级流程

```bash
# 1. 拉取最新代码
git pull origin main

# 2. 备份数据（重要！）
./scripts/full_backup.sh    # 或手动执行备份

# 3. 重新构建镜像
docker compose build

# 4. 执行数据库迁移
docker compose exec api alembic upgrade head

# 5. 重启所有服务
docker compose up -d

# 6. 验证
curl http://localhost:8000/health
```

### 20.2 回滚流程

```bash
# 1. 停止服务
docker compose down

# 2. 切换到旧版本代码
git checkout <previous_tag_or_commit>

# 3. 回滚数据库迁移
docker compose up -d postgres redis
docker compose exec api alembic downgrade -1

# 4. 重新构建旧版本镜像
docker compose build

# 5. 启动所有服务
docker compose up -d

# 6. 如需恢复数据
docker compose exec -T postgres psql -U sguser -d scholarguard < backup.sql
```

### 20.3 零停机升级（高级）

使用 Docker Compose 的滚动更新：

```bash
# 1. 先启动新版本的额外实例
docker compose up -d --no-deps --build api

# 2. 等待新实例健康检查通过
# 3. 旧实例自动替换
```
